# Dark Factory v3.9 (Operative Baseline)

**The accepted-canonical operative baseline of the [[dark-factory]] arc.** v3.9 was accepted by Michael Saucier on **2026-06-05** (`checkpoint-2026-06-05-v3.9-acceptance.md`) as the canonical documentation and specification baseline for Dark Factory implementation. It is derived from the accepted v3.8 baseline plus the **2026-05-13 Civilization Landscape Investigation**, and it preserves v3.8 unchanged as historical evidence while folding the investigation's outputs in as implementation contracts. At the time of compilation it is *the operative implementation baseline*: [[v4-0]] is a later doctrine shift whose seed was accepted 2026-06-12 but which "does not supersede v3.9 until the [v4.0] folder is accepted canonical under its coverage matrix" (*Dark Factory - Motive, Goal, Approach*).

v3.9 is the artifact that turns the [[dark-factory]] thesis — *require every material action to leave evidence in a causal, auditable chain* — into a concrete, layered set of specifications, gates, and obligations. Where [[the-20-primitives]] and the Searles corpus supply the philosophical motive, v3.9 is where that motive becomes a governed software-production architecture.

## What v3.9 is, and what acceptance actually covered

v3.9 is a folder of eleven documents (`README` plus `00`–`09`, plus the acceptance checkpoint). Its self-stated purpose is to "preserve the v3.8 architecture while adding the accepted investigation outputs as implementation contracts, optional adapter boundaries, influence-recording rules, CapabilityArtifact governance, capability-evolution sequencing, operator UX requirements, and native reproducibility requirements" (`DF-V3.9-README`). The version delta from v3.8 is the **U1–U10 update set**.

A distinction the sources are emphatic about, and which is easy to overstate: **acceptance covered the doctrine set only.** The acceptance checkpoint records that the top-level set (`README` + `00`–`09`) is the accepted canonical baseline, and then states explicitly that "subordinate material under `dark-factory/v3.9/implementation/` is **not** part of this acceptance and retains its own `review` / `canonical: false` status until separately accepted" (`DF-V3.9-CKPT-2026-06-05-ACCEPTANCE`). So the *doctrine* is canonical; the *as-built implementation tracker and epic packets* underneath it are evidence, not ratified doctrine. Any claim sourced from the implementation subfolder inherits that lower status.

The acceptance also confirmed four substantive things: U1–U10 are accepted in their stated optional/required/reference roles; **no external framework is promoted** to kernel, truth source, Work replacement, policy owner, release authority, certification authority, capability promoter, factory controller, or Site replacement; v3.8 is preserved unchanged; and open-weight / pi.dev / OpenRouter model routing remains a *future* CapabilityEvolution research topic, not v3.9 base doctrine.

One inheritance rule is called out specifically and is worth not misreading: the `supersedes:` front matter lists `dark-factory/phase-record`, but the README states "`dark-factory/phase-record` is **inherited authority, not superseded** by v3.9" — the Phase 0–3 record and `DF-SOP-0001` protected-side-effect baseline remain binding. The `supersedes:` list must not be read as discarding phase-record authority.

## The base rule: build the pyramid bottom-up

v3.9's organizing constraint is a strict dependency order. "No higher layer may start when its prerequisite control layer is incomplete" (`DF-V3.9-README`). The ordering is:

```text
document governance
  -> EventGraph truth
  -> authority and identity gates
  -> FactoryOrder and Work DAG
  -> bounded runtime
  -> trace completeness
  -> verification and audit
  -> operator review
  -> SaaS generation
  -> memory and knowledge
  -> governed capability evolution
  -> broader autonomy
```

This base rule is the structural expression of the [[dark-factory]] fail-safe posture: control layers must be proven before the capabilities they govern are allowed to run. It mirrors the document set itself, whose nine numbered files map onto these layers.

## The document set (the eight-plus pillars)

The accepted doctrine decomposes into these specifications (`DF-V3.9-README`):

- `00 — document governance`: document rules, semver, front matter, artifact status, deletion criteria.
- `01 — unified architecture decisions`: the consolidated ADR set and rejected alternatives (this is where the "no external framework as kernel" decision lives).
- `02 — kernel schema and state`: the [[event-graph]] kernel, schemas, edges, state machines, projection rebuild rules.
- `03 — authority, identity, and SOPs`: protected actions, identity, lifecycle, approvals, side-effect SOPs.
- `04 — production workflow and runtime`: FactoryOrder, Work DAG, production cells, [[bounded-runtime]], SaaS workflow.
- `05 — verification, audit, risk, eval`: gates, risk register, audit model, eval sufficiency, scorecards.
- `06 — memory, knowledge, capability`: MemPalace, knowledge substrate, capability artifacts, evolution and rollback.
- `07 — technical debt register`: rationalized technical debt and dependency order.
- `08 — implementation workflow checklist`: prescriptive workflow with copy/paste Codex commands.
- `09 — legacy coverage matrix`: explicit coverage of every prior artifact family and baseline obligation.

The orientation doc's twelve-point "Description of Approach" is the readable synthesis of these pillars; the substance below traces each binding rule to the orientation doc, which in turn cites the numbered specs by line. (The wiki lead author has read the orientation doc, the README, the acceptance checkpoint, and the two implementation-state docs directly; the per-line spec citations are the orientation doc's own handles, reproduced for traceability rather than independently re-verified line-by-line this run — see the fail-legible note below.)

### EventGraph as kernel

[[event-graph]] (the `transpara-ai/eventgraph` repo) is the sovereign source of truth for durable facts, causality, audit evidence, identity, authority, lifecycle, artifact provenance, certification, and capability-evolution evidence. Other systems may hold useful state, but **none supersede EventGraph for authority or certification.** The kernel must provide append-only recording, deterministic canonical serialization, tamper evidence, idempotent writes, actor attribution, causal edges, queryable paths, replayable projections, and conformance tests *without live LLM credentials or production side effects* (*Dark Factory - Motive, Goal, Approach*, citing `DF-V3.9-ADR-001` and `DF-V3.9-SPEC-002`).

### Authority and identity gates (fail-closed)

Protected actions are not model suggestions — they require authority, and unknown high-risk actions **deny or require approval**. The canonical authority outcomes are `Autonomous`, `Notify`, `ApprovalRequired`, and `Forbidden`; `ApprovalRequired` blocks until an authorized decision grants approval, and `Forbidden` means the actor must not proceed. Protected examples include production deployment, default-branch pushes, merges to main, secret access, repo create/delete, cross-repo mutation, policy changes, persistent agent spawning, capability promotion/activation/rollback, external runtime invocation, and release certification. This is the [[authority-layer]] made concrete and is the load-bearing fail-closed surface of v3.9.

### FactoryOrder and the Work DAG

A `FactoryOrder` is the durable starting object for product work; it records source intent, requester, scope, constraints, release policy, risk class, and source references. A planner proposes requirements, acceptance criteria, assumptions, ambiguities, options, task drafts, and human-review flags; `transpara-ai/work` turns accepted pieces into tasks and owns the operational DAG — "Work for flow, not truth." Unrelated tests do not satisfy [[acceptance-criteria]], and manual verification is limited by risk class unless explicitly waived.

### Bounded runtime

Runtimes do not govern. They execute only through `RuntimeBroker` envelopes that declare task provenance, actor, authority decision, allowed/denied files, commands, network policy, secrets policy, timeout, resource limits, expected outputs, output contract, trace paths, and post-run validation. The **first runtime is local deterministic execution**; external runtimes are deferred until the local worker proves traceability, policy, and audit. See [[bounded-runtime]].

### Verification and audit

A `ReleaseCandidate` cannot be certified merely because a model, worker, or human says it is good. The `TraceCompletenessGate` blocks certification when FactoryOrder, FactoryRuntimeVersion, acceptance evidence, required gate evidence, certification evidence, or any unresolved high/critical failure is missing. Every product ReleaseCandidate must clear the **product gate minimum** — unit, integration, e2e, build, migration, secret scan, dependency vulnerability/license, SAST, auth-flow and configuration security, container/build-artifact scan where applicable, trace completeness, FactoryRuntimeBOM check, and audit-report check. (Note: the orientation source lists trace completeness, BOM, and audit-report checks *as members of* the product gate minimum, not as a separate tier — see [[gates]] for the same correction.) A proof-of-work packet packages this evidence for human review but does not replace the gates, decisions, receipts, certification, rejection, or audit reports.

### Memory and knowledge are advisory

MemPalace or equivalent recall is advisory — never EventGraph, policy, certification evidence, or authority. A `MemoryReference` or `KnowledgeReference` must be created whenever memory or knowledge materially influences planning, generation, repair, review, or capability evolution. Secret-bearing memory is quarantined and cannot be used; stale knowledge is blocked for high-risk planning; high/critical contradictions block high-risk advisory use.

### Governed capability evolution

Reusable skills, plugins, prompt sections, tool descriptions, workflow packs, schema instructions, evaluation prompts, runtime adapters, and policy bundles are `CapabilityArtifacts` when they can affect factory behavior. Capability evolution flows through `EvolutionOrder -> EvalDataset -> OptimizationRun -> CandidateVariant -> BenchmarkResult -> HumanReview -> CapabilityVersion -> ActivationPolicy -> FactoryRuntimeVersion -> RollbackRecord` (when needed). Optimizers may *propose* improvements but may not merge, activate, bypass benchmarks, weaken security/policy gates, promote their own outputs, or become the factory controller. This is the [[dark-factory]] answer to the "self-improvement circuit breaker" theme in the Searles corpus: self-improvement becomes production work, not an uncontrolled loop.

## The first implementation target: Base Slice 0

The first target is the v3.1 vertical slice, rationalized as **Base Slice 0** — the mandatory control-plane proof before any higher autonomy:

```text
FactoryOrder -> Requirement -> AcceptanceCriterion -> Task -> bounded local worker
  -> Artifact -> TestCase -> TestRun -> GateResult -> TraceCompletenessGate
  -> FactoryRuntimeVersion -> ReleaseCandidate -> Certification or Rejection -> AuditReport
```

Base Slice 0 **must run without** LLM calls, real SaaS generation, external runtime integration, production deploy, multi-agent planning, memory recall, knowledge reuse, or capability self-evolution (`DF-V3.9-README`; `DF-V3.9-SPEC-004` per the orientation doc). It is the floor: prove the evidence chain with the simplest possible mechanism before adding intelligence or side effects.

## Acceptance criteria for v3.9 itself

The README sets the bar for the baseline being acceptable at all: every file has YAML front matter and hidden `df:*` headers; every prior artifact family is represented as standalone content in the folder; every known technical-debt item has an owner surface, prerequisite, status, and next action; every checklist command states whether it targets Codex Desktop or Codex CLI; the base prerequisite workflow forbids higher-layer work until controls are proven; and deletion of prior artifacts would not remove any binding safety, identity, authority, audit, eval, runtime, memory, or capability obligation. The 2026-06-05 checkpoint records these as confirmed.

- **Gate E** — Site governed decision boundary: satisfied **only** for the bounded `transpara-ai/site` decision-surface slice (`site#62`).
- **Gate F** — bounded LLM proposal: satisfied **only** for the bounded `work#38` recorded-LLM-proposal fixture.
- **Gate G** — real scanner orchestration: satisfied **only** for the bounded `work#39` fixture.
- **Gate H** — issue-to-PR proposal autonomy: satisfied **only** for the bounded `work#40` fixture.

- **Epic 10 / Residual Closure** — R-001/R-002/R-003: closed **only** for the bounded `transpara-ai/hive` side-effect-free local-emulation seam (`hive#126`); "real protected side effects remain separately unauthorized."
- **Epic 11** (docs draft-PR live mutation): **selected for authorization review only** — not started — limited to one future `transpara-ai/work` implementation that may create a *draft* PR in `transpara-ai/docs` from an existing `codex/*` branch after the packet is merged, reviewed, and explicitly human-authorized.

So the seed framing "bounded gate completions through Gate J / Epic 10, with Epic 11 selected for authorization review" is accurate **with the qualifier that every one of these is satisfied for a bounded fixture, not as a general production proof.** The repeated phrase across all three sources — "satisfied **only** for the bounded ... fixture" — is load-bearing. Dropping it would assert a production-grade autonomy that the sources explicitly deny. Open Brain corroborates this (2026-06-05): "epics 00–10 all DONE, each bounded to a fixture," and Epic 11 implementation "NOT started."

A second nuance worth flagging: the lettered gates **A–J** here are the v3.9 build-milestone vocabulary and come from the *implementation subfolder + Open Brain*, not from the accepted orientation/README doctrine. They must not be confused with **Gate K / Gate L**, which are v4.0-only future gates (interim dev-loop hardened; v3.9→v4.0 reconciliation certified) and are scoped/partially-planned but **not satisfied** ([[v4-0]], [[gates]]).

## A timing nuance: ratify-after-run vs. ratify-as-doc-act

There is a real tension in the record that the sources resolve in v3.9's favor but worth stating plainly. As of **2026-06-05**, Open Brain captured that v3.9 *doctrine acceptance* was a "PENDING" track **decoupled from implementation** — "you COULD ratify v3.9 canonical today without Epic 11 (Gates A–J done for fixtures)" — and that the honest ordering recommended by demonstration-first philosophy was "run (Slice 1, closes Epic 11) THEN ratify v3.9." The acceptance checkpoint, dated the **same day (2026-06-05)**, records that Michael Saucier did accept v3.9 as canonical. These are consistent: acceptance is described in the checkpoint as covering the *doctrine set*, with implementation evidence (including the still-open Epic 11 live half) explicitly left at `review` status. The article does not treat acceptance of doctrine as a claim that every gate has a *production* proof — the sources are careful to keep those separate, and so is this entry.

## How v3.9 relates to v4.0

[[v4-0]] is a doctrine shift — "the development process itself must become a governed Civilization function: the factory builds the factory under the same accountability it imposes on what it builds." Its seed doctrine (`DF-V4.0-ADR-001`) was accepted by the External Committee on **2026-06-12**, but the orientation doc is unambiguous that "the operative implementation baseline remains Dark Factory v3.9" and that the v4.0 folder/integration program "remains a candidate and is not accepted canonical until the required reconciliation coverage matrix exists." v3.9 stays operative; v4.0's Gate L reconciliation is what would eventually retire v3.9 as the active baseline. Until then, v3.9 is the baseline implementers build against.

## Philosophical basis, held at arm's length

v3.9 inherits the [[dark-factory]] lineage from the Matt Searles corpus: traceability becomes [[event-graph]]; explicit criteria become [[acceptance-criteria]] and [[gates]]; agent action becomes [[bounded-runtime]] invocation; authority becomes AuthorityRequest/AuthorityDecision/ExecutionReceipt; self-improvement becomes the governed EvolutionOrder loop. But v3.9's own acceptance criteria carry forward the corpus's guardrail rather than its certainty. The Searles "Cult Test" post (2026-02-28) is explicit that **"the framework is a tool, not a truth," "the scientific method is the authority layer,"** and the framework "must not become sacred" ([[the-cult-test]], [Searles-Cult-Test]). [[the-20-primitives]] are adopted in Dark Factory as *accepted philosophical basis only* — conceptual grounding for EventGraph, FactoryOrder, criteria, and bounded agents — never as proven metaphysics or implementation authority. v3.9's "What Dark Factory Is Not" list ends on exactly this point: it is not "a claim that philosophical primitives are proven metaphysics."
