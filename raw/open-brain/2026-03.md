## 2026-03-03 · observation · (recovered via search)

Critical crane pattern for tinstaller: crane pull --format=tarball is REQUIRED. Without it, crane produces OCI layout tars with NO RepoTags in manifest.json. When k3s ctr images import reads these, images are imported as untagged sha256 digests — this was THE root cause of cert-manager/prometheus/CNPG ImagePullBackOff in air-gap mode. Fixed in commit 430b728. Also: k3s ctr images import rejects tar-of-tars (use individual .tar per image), and k3s auto-import only works at startup (files placed in /var/lib/rancher/k3s/agent/images/ after K3s starts are NOT auto-imported).

Transpara Platform — Specification-Driven Development Workflow

The transpara/platform repo is the specification layer between human intent systems (Confluence, Jira, Figma) and agent execution systems (Claude Code, Claude Desktop, CI/CD).

Workflow: SPECIFY → PLAN → GENERATE → REVIEW → VERIFY → CLOSE

Agent roles:
- Specify: Claude Desktop reads Confluence/Jira/Figma, writes specs/features/*.md
- Plan: Claude Desktop reads spec + Platform Spec + ADRs, updates Implementation Plan
- Generate: Claude Code reads spec on disk, generates code in target repo, creates PR
- Review: Claude in Chrome assists with browser-based PR review
- Verify: Claude Code checks cluster state against spec verification criteria
- Close: Claude Desktop updates spec status → Shipped, updates capability matrix

- Claude Desktop (Windows): reads via GitHub MCP + Atlassian MCP + Figma MCP, writes via github:create_or_update_file

## 2026-03-04 · observation · (recovered via search)

Transpara Platform — Spegel P2P mechanics and containerd GC resilience (Air-Gap Operations v1.0.1, Sections 4.3–4.4, 4.7).

**Spegel P2P (K3s v1.35+):** Embedded stateless OCI mirror. Enable with --embedded-registry + --disable-default-registry-endpoint. Workflow: (1) K3s starts with Spegel flags → P2P DHT initialized, (2) install-tsystem on node 1 pushes images to crane (127.0.0.1:5111), (3) Join nodes inherit registries.yaml mirror config pointing to crane, (4) Spegel discovers images via DHT, serves them via P2P to other nodes, (5) Join nodes do NOT need --images-only — Spegel distributes images automatically. Validated: T13, T14 (3-node air-gap) pass without --images-only on join nodes.

**PushContainerdImagesToCrane — K3s system image GC resilience:** Problem: Spegel mode with --disable-default-registry-endpoint disables upstream pulls. All pulls go through crane. Containerd GC can evict K3s system images (pause, coredns, metrics-server) not referenced by running pods. Without these in crane, pod creation fails with no recovery. Solution (pkg/airgap/images.go): k3s ctr images list -q → filter for rancher/*, registry.k8s.io/*, envoyproxy/* → k3s ctr images export --platform linux/amd64 <tmp.tar> <ref> → crane push to 127.0.0.1:5111. Idempotent via crane digest check before each export+push.

**Containerd GC mitigations (all implemented):** (1) Per-category tar cleanup: CleanupTars() removes .tar files after each category pushed to crane. (2) Temp file cleanup: defer os.Remove() for extracted charts/YAML. (3) Disk pressure detection: CheckDiskSpace() warns if <10GB available. (4) Crane registry persistence: images in crane survive containerd GC — pods re-pull from 127.0.0.1:5111.

**.cache.json handling (resolved):** K3s writes .cache.json to track imported tarballs in /var/lib/rancher/k3s/agent/images/. If containerd loses images but cache exists, K3s skips reimport. Fix: PlaceArtifacts() removes stale .cache.json before every install. VerifyAirgapImages() polls for pause image after K3s starts — if missing, deletes cache and restarts K3s.

## 2026-03-04 · observation · (recovered via search)

Transpara — KPI status model, terminology, and key product concepts (RTOI v8).

KPI Status Model:
- A metric becomes a KPI when at least one limit is defined (giving it status).
- Limit hierarchy (low to high): LLL (Low Low Low), LL (Low Low), L (Low), Good (no limit crossed), H (High), HH (High High), HHH (High High High). All renamable.
- Limits can be static, dynamic, or calculated.
- Min/Max: Engineering limits — values beyond these indicate broken data (Out of Range status).
- Target: Separate from limits — the ideal/desired value.

Additional statuses beyond limits:
- NA: No limits defined (absence of status)
- Out of Range: Value beyond min or max
- Not in Service: Override status for maintenance or non-normal operation (manual or dynamic)
- Stale: Data hasn't arrived within configured freshness threshold (e.g., 15 min for 1-min data)
- Custom: User-defined digital states (on/off, running/paused/stopped)

Key terminology distinctions:
- Element (formerly "Group"): Logical grouping of KPIs representing asset, site, area, department, process
- Interface: Read live data from external source without storing
- Extractor: Extract data from source into tStore (for storage/caching)
- RCS: Remote Context Service — sync metadata from external sources (AF, inmation, VMware, Zabbix)
- Managed visualization: Admin-created, governed, centrally managed
- Ad-hoc visualization: User-created on the fly, non-destructive, temporary unless saved
- Time Windows (batch/shift): Event-based time segmentation for calculations and analysis
- Templates: Multi-level reusable configurations — from individual KPI to entire site/plant (unique to Transpara)

Traditional "prompting" has fractured into four cumulative disciplines. Each builds on the one below:

1. PROMPT CRAFT (Foundation) — Structured request formulation. Single-turn or short multi-turn. Session-scoped, no persistence. Output quality depends on operator's ability to articulate in the moment. Hits the "35-minute wall" — iterative refinement exceeds spec-writing cost at ~35 minutes.

2. CONTEXT ENGINEERING — Persistent knowledge environments that load institutional context into every AI session. Architectural, not conversational. Transpara artifacts: CLAUDE.md hierarchy, .claude/instructions.md, ~/.claude/CLAUDE.md, MEMORY.md + topic files, MCP servers (Confluence, Jira, Figma, GitHub), Information Source Hierarchy. The transpara/platform repo itself is a context engineering artifact.

3. INTENT ENGINEERING — Encoding decision-making rules, tradeoff hierarchies, and judgment frameworks. Addresses "smart-but-wrong" failure: technically correct outputs that miss the purpose. Transpara artifacts: ADRs (26 accepted), conflict resolution rules (code > spec > Confluence > Jira), authority levels (Normative > Prescriptive > Reference), session scoping (Deep/Tactical/Exploratory), Platform Key Rules, finding severities. The Klarna lesson: context without intent optimizes for the wrong outcome.

4. SPECIFICATION ENGINEERING (Apex) — Autonomous execution contracts: acceptance criteria, constraint architecture, task decomposition, evaluation criteria. The spec is the complete contract — agent executes without further human input. Enables 10x productivity over conversational prompting. Transpara artifacts: specs/features/*.md, specs/_template.md, capabilities/matrix.md, Platform Specification invariants.

Cumulative principle: Cannot write effective specifications without encoding intent. Cannot encode intent without providing context. Cannot provide context without basic prompt craft.

Transpara maturity: L1 Prompt Craft (practiced), L2 Context Engineering (implemented), L3 Intent Engineering (implemented), L4 Specification Engineering (emerging — specs/ exists with 20+ features, not yet default for all new work).

L4 completion criteria: All new work starts with spec, specs include constraint architecture, acceptance criteria verifiable without judgment, agents execute Steps 2-6 autonomously, capability matrix current and linked, cross-environment handoff works, time-to-implementation measurably lower.

Transpara Platform — Session scoping model: Deep, Tactical, Exploratory (Project Instructions v0.9.5, Section 1).

Three scope levels calibrate depth and output format at session start:

Deep scope: Platform architecture, multi-stakeholder specs, security-critical design, adversarial cross-review. Produces versioned documents with numbered invariants, verification gates, compliance mapping, explicit rationale. Full treatment — no shortcuts, no truncation. Effort estimates included.

Tactical scope: Single deliverable, implementation task, config generation, focused analysis, opinion request. Produces single file, code artifact, or inline answer. Direct and efficient — skip ceremony. Brief or omitted rationale.

Exploratory scope: Unclear requirements, brainstorming, research, "help me think about X." Conversation first, artifacts if/when direction solidifies. Collaborative — ask clarifying questions, iterate in chat before committing to structure.

How to establish: If obvious, state assumed scope and proceed. If ambiguous, present three options — one exchange. Scope can escalate mid-session: "This is turning into a deeper design question — want to escalate to deep scope?" Presets are defaults, not cages.
