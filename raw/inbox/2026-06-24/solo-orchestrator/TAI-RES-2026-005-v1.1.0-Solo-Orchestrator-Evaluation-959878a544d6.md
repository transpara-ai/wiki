## Abstract

This document evaluates **Solo Orchestrator** — the AI-assisted, phase-gated software-development *methodology* at `transpara-ai/solo-orchestrator` (a fork of the public `kraulerson/solo-orchestrator`), taken at HEAD `a4a9ab8` (2026-05-17, release **v1.0.0**, MIT) [1] — and compares it against the Transpara-AI Civilization architecture. It is the fifth in the external-landscape evaluation series, applying the same pattern used for Sakana [20], Hermes [21], OKF [22], and MemPalace [23]. Unlike the prior Solo Orchestrator treatment — an investigation-tier wiki article that, by its own admission, *"inspected only the README"* and marked every claim about the project's internals *"thin / README-grounded, not code-verified"* [17] — this evaluation read the **74 shell scripts and 24 templates** as ground truth.

The central finding is that Solo Orchestrator is the most **methodologically convergent** external artifact the Civilization has evaluated. It independently arrived at almost exactly the Civilization's implementation discipline — five phase gates (Discovery → Architecture → Construction → Validation → Release), test-driven and documentation-mandatory, with ADRs, approval logs, security scans, proof-of-work UAT, rollback tests, and even vendored agent *skills* [1][2]. But the two systems sit on opposite sides of a single seam, and code reading is what reveals it: **Solo Orchestrator expresses its gates as local, advisory, mutable documents enforced by a Claude-Code session hook** — fail-closed as a speed-bump against the in-session AI, but **fail-open at the trust boundary** (a human at a raw terminal, a `SOIF_PHASE_GATES=warn`, a hook removed from `.claude/settings.json`, or a direct edit of `.claude/phase-state.json` walks straight past every gate; the actual phase gate is not even wired to a hook; the artifacts are existence-checked Markdown the framework itself concedes are *"not sufficient evidence of approval"*) [2][3]. The Civilization makes the **same gates fail-closed predicates at the EventGraph store boundary** — certification cannot write a `Certification` record unless every gate's evidence record exists, EventGraph is the sovereign source of truth, and *"Git, CI, branches, PRs, and worker outputs may hold useful state, but none supersede EventGraph for certification or authority"* [11][12].

The one-line result: **Solo Orchestrator writes the gates down as documents a Claude session enforces; the Civilization makes the same gates records the kernel cannot certify around.** This is precisely the disposition the 2026-05-13 landscape investigation already ratified — *"workflow/operator UX/proof-of-work references only … no Work/Site/controller replacement"* [14] — and its load-bearing objection, *"Markdown/process state not EventGraph,"* is now code-confirmed rather than README-inferred [17]. Six concrete learning opportunities follow, all consistent with that pattern-only decision and Decision 15 [13]: they describe how to *mine Solo Orchestrator's concrete gate implementations and translate them into EventGraph records*, never how to run it as a runtime, control plane, or authority.

## 2. Solo Orchestrator: Capability Overview

Solo Orchestrator describes itself as *"not vibe coding … a phase-gated, test-driven, documentation-mandatory process with security scanning, threat modeling, and incident response built in,"* for *"a single technically literate person building MVP-grade applications using AI as the execution layer"* [1]. It is implemented as ~74 shell scripts, 24 Markdown `.tmpl` templates, JSON state files, Claude Code hooks/skills, a UAT harness, and CI/release templates for *generated* projects — there is no application code; the "product" is the methodology and the scripts that check it.

> ⚠ **Version note (closing a prior open item).** The wiki flagged conflicting *"v4.1"* vs *"v1.0"* labels [17]. Code-verified: Solo Orchestrator is **v1.0.0** (`init.sh` `VERSION="1.0.0"`; every README status row says v1.0). The *"v4.1"* is a **separate dependency** — the `claude-dev-framework` ("Development Guardrails," v4.x) that `init.sh` clones from the same author. There is no "Solo Orchestrator v4.1." [5]

### 2.1 The Five Phases (Spec-Anchored)

Each phase has gate-checked exit artifacts (verified against `check-phase-gate.sh`) [2][3]:

| Phase | Required exit artifacts | Approver |
|---|---|---|
| 0 Discovery | `PRODUCT_MANIFESTO.md` (8 sections); `frd`/`user-journey`/`data-contract` | Project Sponsor |
| 1 Architecture | `PROJECT_BIBLE.md` (≥14 of 16 sections); ADR-0001 (architecture) | Senior Technical Authority |
| 2 Construction | `FEATURES.md`, `CHANGELOG.md`; bug gate (no open SEV-1/2); per-feature Build Loop | Orchestrator / Sr Tech Authority |
| 3 Validation | test-results, `sbom.json`, `HANDOFF.md`, `INCIDENT_RESPONSE.md`, `SECURITY.md`, pen-test (Full track) | Application Owner + IT Security |
| 4 Release | `RELEASE_NOTES.md`, rollback/handoff test evidence | App Owner + IT Security |

### 2.2 The Gate Model — Fail-Closed Against the Agent, Fail-Open at the Trust Boundary (Code-Anchored)

This is the question the README-only investigation could not answer. Reading the scripts gives a precise, two-sided answer [2]:

- **Genuinely BLOCKING — but only inside a Claude Code session.** `pre-commit-gate.sh` and `session-mcp-gate.sh` emit real Claude Code `permissionDecision:"deny"` JSON; they enforce the Build Loop, UAT completeness, phase-3/4 step completion, and string-bans on `--no-verify`/`--force`/`--reset`. **Their authority is local:** they are `PreToolUse` hooks registered in the project's own `.claude/settings.json` (`init.sh:1446-1544`). Anyone committing via a plain terminal, another tool, or after deleting the hook entry is **not gated at all** — there is no server-side or CI enforcement of them.
- **The actual phase gate is advisory in practice.** `check-phase-gate.sh` is **not wired into any hook**; it runs only if a CI workflow or a human invokes it — and `init.sh` installs **no CI**. Even when run, **`SOIF_PHASE_GATES=warn` downgrades every block to exit 0**, most checks are WARN-level, and it parses a hand-editable JSON via `grep`.
- **State is a mutable working file, not a durable record.** Phase/gate state lives in `.claude/phase-state.json`, `process-state.json`, `build-progress.json` — read and *written* by the very scripts the agent drives (`--complete-step`, `--record-feature` mutate them). Nothing is hash-chained or signed; the README itself says these files *"mutate as the project progresses."* Gate state is whatever the last writer (agent or human) put in the JSON.
- **Two genuinely fail-closed controls exist, and both echo the Civilization's own instincts** — narrow but real: (1) **self-approval detection** — `check-phase-gate.sh:178-191` FAILs if the approver name in `APPROVAL_LOG.md` equals `git config user.name` (the *"actor cannot approve its own work"* principle); (2) **real secret/SAST scanning at commit** — `verify-install.sh:680-704` installs a git pre-commit hook that runs `gitleaks` and `semgrep scan --config=p/owasp-top-ten` and blocks on findings, and `pre-commit-gate.sh:86` denies `git commit --no-verify`. Both run only if the binaries are installed and the human commits through the hook.

> ⚠ **Evidence note.** §2.2 is **code-verified** (every exit code and deny-JSON read line-by-line). The prior README-only investigation would have seen *"mechanical enforcement / no skipping ahead"*; the code shows that holds **only against an in-session agent**, not against a human, a raw terminal, `SOIF_PHASE_GATES=warn`, or a direct state edit. The framework's own docs concede this: phase gates *"rely on the AI agent following CLAUDE.md … Tier 3 (guided) controls with no automated backstop yet,"* and *"only the CI pipeline is a hard enforcement boundary"* — a CI pipeline it ships as a template but does not install.

### 2.3 Artifacts, ADRs, and Governance (Code-Anchored)

The mandated artifacts (product-manifesto, project-bible, ADRs, approval logs, threat-model, security-audit, incident-response, handoff, rollback-test) are **documentation a human or agent fills in**, gated only on **existence and shape** (file present, N numbered headings, no `[placeholder]`/`YYYY-MM-DD` literals) — never on the *truth* of the content [3]. The framework's own governance doc states it plainly: *"The existence of an artifact alone is not sufficient evidence of approval"* — yet the tooling enforces exactly and only existence.

On **ADRs** (specifically flagged for this review): Solo Orchestrator treats ADRs as a first-class artifact (a Nygard-style `adr.tmpl`: Status / Date / Phase / Supersedes / Context / Options / Decision / Rejected Alternatives / Consequences), mandated *in prose* for Phase 1 architecture and any in-flight Phase-2 change [3]. But **no gate script references ADRs at all** — an exhaustive grep of `scripts/` and `tests/` finds zero enforcement-path references. Nothing fails if an ADR is absent, stale, or contradicted by the code, and there is **no machine-readable link from any ADR to a test, gate, or CI check**. Approval logs are the same shape: a typed Markdown table whose `Reference`/`Method` columns point to off-system evidence (an email, a ticket #) the framework cannot follow or verify; the only tamper-evidence is the git-author-equals-approver convention.

### 2.4 Tooling, Skills, Memory (Code-Anchored)

- **Agent-agnostic docs, Claude-Code-wired automation.** The README's two-layer framing is accurate: the *methodology* (phases, templates, governance prose) is portable, but the entire *automation layer* — seven hook scripts speaking Claude Code's stdin-JSON protocol — is hard-wired to Claude Code. *"Everything that makes it autonomous is Claude-Code-only"*; without it, all mechanical enforcement goes dark [5].
- **Vendored agent skills.** It ships four `SKILL.md` skills (`zoom-out`, `session-handoff`, `grill-with-docs` — adapted from `mattpocock/skills`, MIT-attributed; `sweep-triage` — original) installed to `.claude/skills/`. It *recommends* (does not bundle) the separate Superpowers plugin [5].
- **Memory** is an **optional Qdrant MCP**, advisory by default (`session-end-qdrant-reminder.sh` nags but does not block), but convertible to a hard `PreToolUse` deny-gate if `qdrant_required=true` [5].
- **Sessions** persist as `.claude/*.json` + `CLAUDE.md` + git, resurfaced across runs via a **copy-paste prompt** that `resume.sh` prints for the human to paste back into Claude Code [5].

### 2.5 Supply Chain & Maturity

MIT, *"Copyright (c) 2026 Karl Raulerson"* [4]. The transpara-ai fork is a **clean vendored mirror** of upstream (`gh` compare: `status: identical`, 0 ahead/0 behind), so there is no Transpara divergence to maintain. Upstream is an **obscure single-maintainer project**: created 2026-04-02 (~2.5 months old), **20★ / 5 forks**, **328 commits by one author**, **0 releases/tags**, **no repo-level CI** (CI exists only as templates for generated projects), 37 shell test files, last commit 2026-05-17 (~5 weeks stale). Bus factor 1 — which the README's own *"backup maintainer model"* prescribes but the project does not itself have [24]. Because Solo Orchestrator is admitted **pattern-only** (its *code* never runs in the factory), this maturity profile is a lower-stakes concern than it was for an integration candidate like MemPalace [23] — we mine its patterns, we do not depend on its scripts.

### 2.6 Competitive Context (Third-Party-Cited)

Solo Orchestrator's *ideas* are the mid-2026 consensus, not novelty. Phase-gated, spec-driven, TDD-mandatory, security-built-in AI development is the field's reaction to "vibe coding," and far more adopted peers occupy it: **GitHub Spec Kit** (~90k★), **BMAD-METHOD** (~49k★), **AWS Kiro**, Harper Reed's workflow, Anthropic's Claude Code best-practices; its security/TDD rigor mirrors active academic threads (Constitutional SDD, TDFlow) rather than leading them [25]. Solo Orchestrator itself is an **obscure personal project** — ~20★, single author, **not listed in the principal 30+-framework survey, with no Hacker News / Reddit / independent coverage** found [25]. Its one modestly distinctive angle is *"enterprise-grade governance (approval authorities, compliance, backup-maintainer model) for a solo builder."* The evaluation implication: Solo Orchestrator is not interesting as a *methodology* (the patterns are everywhere) — its value to the Civilization is its **concrete, code-level gate and artifact implementations**, which are unusually complete for a project this size.

> ⚠ **Evidence note.** §2.6 is **third-party-cited** (vendor pages, an independent landscape survey, the GitHub API) and carried at lower confidence; no external assessment of Solo Orchestrator *specifically* exists [25].

## 3. Transpara-AI Civilization: Baseline Architecture

The Transpara-AI Civilization is a self-organizing AI agent civilisation built on the EventGraph substrate. (This section reuses the TAI-RES-2026-001/-002/-003/-004 baseline [19] so the five evaluations stay parallel, and foregrounds the **production-workflow and gate machinery**, which is the native counterpart of a phase-gate methodology.)

### 3.1 Architecture Layers

| Layer | Description |
|---|---|
| EventGraph Substrate | 201 primitives. Causal chain where every action, resource, and outcome is linked. Trust model, authority levels, cryptographic consent. On Postgres on-prem. **The sovereign source of truth (§3.3).** |
| Hive Runtime | 4 bootstrap agents (CTO, Guardian, Researcher, Planner/Implementer). FSM-based agentic loop. Go binary. |
| Social Grammar | Constitutional law for agents. 10 invariants. 8 formal agent rights. Dual human-agent consent to amend. |
| **Production Workflow** | Gate-first factory production: `FactoryOrder → … → GateResult → TraceCompletenessGate → ReleaseCandidate → Certification/Rejection → AuditReport`. The Release cell is the sole certification authority. Detailed in §3.2. |
| Capability Evolution Chain | Governed self-improvement — the subject of the Hermes evaluation [21]. |
| 13 Product Layers | Work, Market, Social, Justice, Build, Knowledge, Alignment, Identity, Bond, Belonging, Meaning, Evolution, Being. |
| Self-Sustaining Economy | Revenue funds agents; agents build products; every resource on the causal chain, publicly auditable. |

### 3.2 The Gate-First Production Workflow (the native counterpart of a phase-gate methodology)

The factory's production discipline is `DF-V3.9-SPEC-004` [10]. Base Slice 0 is a **mandatory deterministic gate DAG**: `FactoryOrder → Requirement → AcceptanceCriterion → Task → local deterministic worker → Artifact → TestCase → TestRun → GateResult → TraceCompletenessGate → FactoryRuntimeVersion → ReleaseCandidate → Certification or Rejection → AuditReport`. The Release cell is the *only* cell that may certify, *"and only through `release.certify` authority and TraceCompletenessGate evidence"* — and certification requires *"actor can certify, trace passes, gates pass/waived, BOM linked,"* failing closed on `trace_incomplete`, `missing_gate_evidence`, or `unresolved_critical_failure`. The **implementation-workflow checklist** `DF-V3.9-PLAN-008` carries a *universal closeout gate* — every step must record validation commands, CI status, the adversarial-review verdict, and residual-risk links in the PR closeout [10]. This checklist (`08`) is exactly the document the prior investigation said Solo Orchestrator's gates should map onto [17].

### 3.3 EventGraph as the Sovereign Record (the seam this evaluation turns on)

The Civilization's load-bearing principle — and the direct refutation of "Markdown/JSON process state as authority" — is `DF-V3.9-ADR-001` Decision 1 [11]:

> *"EventGraph is the sovereign source of truth for durable facts, causality, audit evidence, identity records, authority records, lifecycle records, artifact provenance, certification evidence … Work, Site, Hive, Git, CI, MemPalace, LLM Wiki, runtime logs, branches, PRs, and worker outputs may hold useful state, but **none supersede EventGraph for certification or authority**."*

The source-of-truth hierarchy is explicit that *"agent messages and chat transcripts are non-authoritative unless converted into EventGraph records,"* the invariants forbid *"hidden state that cannot be audited or rebuilt,"* and the `eventgraph` kernel states it in one line: *"RECORD — the event graph is the source of truth. If it isn't recorded, it didn't happen"* [11]. The enforcement is fail-closed **in code**: the authority chain `AuthorityRequest → AuthorityDecision → ExecutionReceipt` is defined as first-class types (`schema.go` L371, L385, L397), `AuthorityDecision.Validate` is an **allowlist** (`validate.go:485` — `requireOneOf("Autonomous","Notify","ApprovalRequired","Forbidden")`, no permissive fall-through), the store boundary validates-before-write and is append-only/immutable (`store.go:40-79`), and `CertifyReleaseCandidate` refuses to write a `Certification` unless `EvaluateCertificationEligibility` returns a **conjunction** of all gate paths passing — returning `ErrRequiredPathMissing` on any gap (`release.go:53-102`, `query.go:376-414`) [12]. In the baseline agent's words, *"certification is the explicitly-proven branch; every gap returns an error rather than proceeding"* — the Civilization's gate is an EventGraph record the kernel verifies, not a checkbox in a side file.

## 4. Comparative Analysis

The central finding is that Solo Orchestrator and the Civilization are **complementary at the production-discipline layer they most converge on** — but they sit on opposite sides of one seam: Solo Orchestrator expresses its gates as *local, advisory, mutable documents enforced by a Claude session* (fail-open at the trust boundary); the Civilization owns the *sovereign, fail-closed, kernel-enforced* version of the same gates. Solo Orchestrator is, gate for gate, a reference *methodology* for the discipline the Civilization already enforces in its substrate.

### 4.2 Shared Territory

Solo Orchestrator converges on the Civilization's instincts more completely than any prior subject — it is the same gate-first doctrine, independently built:

- **Phase gates before progress.** Solo's five phases and the Civilization's `FactoryOrder → … → Certification` DAG are the same idea: no step proceeds until its evidence exists [1][10].
- **Test-driven, documentation-mandatory.** Solo mandates TDD, ADRs, and a Project Bible; the Civilization's closeout gate mandates validation commands, tests, and adversarial-review residue in every PR [1][10].
- **Proof-of-work and approvals.** Solo's UAT reports, approval logs, and rollback tests are the Civilization's proof-of-work surfaces and `HumanApproval`/audit records in different clothing [3][12].
- **Actor cannot approve its own work.** Solo's self-approval detection (`check-phase-gate.sh:178-191`) is the same instinct as the Civilization's *"optimizer cannot promote its own output"* [3][21].
- **Capabilities as governed artifacts.** Solo vendors agent skills; the Civilization treats skills/plugins as CapabilityArtifacts deserving provenance — both independently chose to treat agent capabilities as first-class [5][21].

### 4.3 The Civilization's Structural Advantages

These have no Solo Orchestrator equivalent and represent the deepest difference — every one lives in the *sovereign-substrate* half of the seam:

- **The gate is a record, not a checkbox.** Solo's gate state is a mutable JSON file a hook reads; the Civilization's gate is an immutable, validated EventGraph record certification cannot pass without [11][12]. This is the difference between a checklist and an audit trail.
- **Enforcement is at the substrate, not the session.** Solo's strongest gates are local Claude-Code hooks any human or tool can bypass; the Civilization's gates are predicates at the store/append boundary that no actor can route around [2][12].
- **One sovereign source of truth.** Solo runs a *parallel* Markdown/JSON authority state; the Civilization forbids exactly this — *"none supersede EventGraph for certification or authority"* [11]. A parallel authority state is the precise failure mode the kernel exists to prevent.
- **Agent-independence.** Solo's automation is non-functional without Claude Code; the Civilization's gates bind any actor that writes to the kernel [5][12].
- Plus the civilisation-wide advantages shared with the prior evaluations: constitutional agent rights, the self-sustaining economy, and 13-layer scope [20][21].

Its only ratified disposition is the grouped Legacy Coverage Matrix L257 under Decision 15 [13][14].

## 5. Strategic Learning Opportunities

Six concrete, implementable opportunities. None requires running Solo Orchestrator's code: each mines a *pattern* and translates it into the Civilization's fail-closed substrate, consistent with the pattern-only decision and Decision 15 [13][14].

### 5.1 Solo's Phase-Gate Artifacts → EventGraph `GateResult` Records (translate, don't transcribe)

**Current state:** The implementation-workflow checklist (`08`) specifies gates; Solo Orchestrator has the most complete *concrete* gate-artifact set in the surveyed landscape [10][3].

**Solo pattern:** Five phases with defined exit artifacts, checked by `check-phase-gate.sh` — but as Markdown existence checks over a mutable JSON state [2].

**Learning:** This is the load-bearing, *fail-closed* learning. Mine Solo's concrete gate definitions (the per-phase required-artifact lists) into the `08` checklist, but emit each gate as a **`GateResult` / `TraceCompletenessGate` evidence record** on EventGraph — never a Markdown checkbox or a `.claude/*.json` flag. A phase gate becomes evidence the kernel verifies before `CertifyReleaseCandidate` can pass [12], not a file whose existence a session hook greps. The translation rule the investigation already set stands: *gates become records the kernel can verify, not checkboxes in a side file* [17].

### 5.2 Solo's Self-Approval Detection → A Fail-Closed Predicate on `HumanApproval`/`AuthorityDecision`

**Current state:** The authority chain validates decision enums and requires an active decider, but has no explicit "approver ≠ actor" predicate surfaced in this review [12].

**Solo pattern:** `check-phase-gate.sh:178-191` FAILs when the approver name equals `git config user.name` — a real fail-closed control [3].

**Learning:** Adopt the principle as a code-enforced predicate: a `HumanApproval`/`AuthorityDecision` whose decider is the same actor that produced the work is rejected at validation — the same shape as the Capability Evolution chain's *"optimizer cannot promote its own output"* [21]. Cheap, additive, and squarely in the fail-closed-by-default standard.

### 5.3 Solo's Commit-Time Scanners → `GateResult` Evidence, Not a Templated Audit

**Current state:** The security cell and Base-Slice-0 gates require security evidence; the *form* of that evidence is unspecified [10].

**Solo pattern:** `verify-install.sh` runs real `gitleaks` + `semgrep` at commit and blocks on findings — but its *audit artifacts* are filled-in Markdown a human types, gated only on file existence [2][3].

**Learning:** Take the strength (scanners actually run) and invert the weakness (results as prose): a scan run emits a **`GateResult` evidence record** (tool, config, findings, pass/fail) that the security cell consumes and certification checks — never a `docs/security-audits/*.md` whose existence stands in for a scan. Solo Orchestrator is the worked example of why scan *output*, not a scan *document*, must be the gate.

### 5.4 Solo's Template/Artifact Menu → The Proof-of-Work Record Set (reference-only)

**Current state:** The `08` checklist names proof-of-work and operator-review surfaces (D0/D0b) but not a concrete artifact catalogue [10].

**Solo pattern:** 24 mature templates — ADR, project-bible, threat-model-validation, data-contract, approval-log, handoff, rollback-test, release-notes — and a UAT harness [3].

**Learning:** Use Solo's template set as a **concrete checklist** for which proof-of-work records the factory should produce, each rendered from (and linked on) EventGraph rather than left as a standalone Markdown file. This is reference-only adoption — exactly the "mine UX, gates, proof-of-work, and workflow patterns" boundary the matrix already grants [14].

### 5.5 Solo Orchestrator → The Doctrine's Worked Example of "Session Hook vs Sovereign Record"

**Current state:** Decision 1 forbids a parallel authority state, but the doctrine has no concrete external illustration [11].

**Solo pattern:** A well-engineered methodology whose gates are nonetheless fail-open at the trust boundary — bypassable by a raw terminal, `SOIF_PHASE_GATES=warn`, a hook removal, or a JSON edit, with state in a mutable side file [2].

**Learning:** Write Solo Orchestrator into the knowledge doctrine as the **canonical case study** of why gates must live in the sovereign record, not in a session hook over Markdown/JSON. It makes Decision 1's abstract prohibition concrete, and pre-empts the failure mode of an agent-local checklist masquerading as governance.

### 5.6 Close the Decision-Record Gap (docs hygiene)

**Current state:** Solo Orchestrator alone among its five matrix-siblings has **no per-item crosswalk row** [15][17].

**Solo pattern:** n/a — this is an internal bookkeeping gap.

**Learning:** Give Solo Orchestrator a per-item crosswalk row for parity, recording the now-code-verified facts (v1.0.0; pattern-only; README-only→code-verified; fork 0-divergence; the "Markdown-state-not-EventGraph" objection confirmed). This is documentation hygiene, **pattern-only status unchanged**; no dedicated ADR is opened unless §5.1's gate-translation is taken up as real work.

## 7. Conclusions

Solo Orchestrator is the most *methodologically convergent* external system the Civilization has evaluated: phase gates, TDD, mandatory docs and ADRs, approval logs, security scans, proof-of-work UAT, and vendored agent skills — independently arrived at, and unusually complete for an obscure solo project. Reading the code (the decisive advance over the README-only prior investigation) shows the convergence is real and so is the seam: Solo Orchestrator's gates are **local, advisory, mutable documents a Claude session enforces** — fail-closed against the in-session agent, but **fail-open at the trust boundary**, with a parallel Markdown/JSON authority state the framework itself admits is *"not sufficient evidence of approval."* The Civilization makes the **same gates fail-closed records at the EventGraph store boundary**, where certification cannot pass until the evidence exists and *nothing supersedes the sovereign record*.

The difference is therefore not the gate; it is **where the gate lives, and whether it can be skipped.** Solo Orchestrator puts the gate in a session hook over a side file; the Civilization put it in the substrate, as a predicate no actor can route around. That confirms — now with code, not a README — exactly the disposition the landscape investigation ratified: **mine the gates, the proof-of-work, and the workflow patterns; cede no authority.**

**Priority recommendations:**

1. **Highest priority:** Translate Solo's concrete phase-gate artifacts into **`GateResult` EventGraph records** in the `08` checklist (§5.1) — a gate is a verified record, never a Markdown checkbox or a `.claude/*.json` flag.
2. **High priority:** Port Solo's self-approval-detection control into a fail-closed **"approver ≠ actor"** predicate on `HumanApproval`/`AuthorityDecision` (§5.2).
3. **Medium priority:** Make security-scan *output* a `GateResult` record, not a templated audit doc (§5.3); mine Solo's template menu as the proof-of-work record set (§5.4).
4. **Medium priority:** Write Solo Orchestrator into the doctrine as the worked example of "session hook vs sovereign record" (§5.5).
5. **Lower priority:** Close the per-item crosswalk-row gap for parity, pattern-only status unchanged (§5.6).

**Standing constraints reaffirmed:** Per Decision 15 and Legacy Coverage Matrix L257, Solo Orchestrator stays a **workflow/proof-of-work pattern reference** — never a runtime, control plane, Work/Site/controller replacement, or authority. Its *code never runs in the factory*; only its patterns are mined, and every mined gate is translated into a sovereign EventGraph record rather than left as Markdown/JSON authority state.
