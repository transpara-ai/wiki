## Abstract

This document evaluates the **Hermes Agent** fork family (`transpara-ai/hermes-agent`, `hermes-agent-self-evolution`, `hermes-example-plugins`, forked from the public `NousResearch` upstream) and compares it against the Transpara-AI Civilization architecture. It is the companion analysis to TAI-RES-2026-001 (Sakana AI), applying the same evaluation pattern to a different subject.

Two findings distinguish this evaluation from the Sakana one. First, the two systems are again **complementary rather than directly comparable**: Hermes operates at the *agent-runtime and offline-optimizer* layer (a self-improving agent plus a DSPy+GEPA pipeline that evolves that agent's skills, prompts, tools, and code), while the Civilization operates at the *governance, substrate, and societal* layer. Second — and unlike Sakana, which is a distant company — **Hermes has already been pulled into the transpara-ai org, read as source, and formally adjudicated.** The 2026-05-13 Civilization Landscape Investigation classified it *"governed optimizer pattern only / deferred,"* ADR-0010 proposed it as a *governed capability optimizer*, and it credited two native update items (U7 CapabilityArtifact governance, U8 CapabilityEvolution sequencing). The native EventGraph **Capability Evolution chain** that resulted was then built and adversarially hardened in code.

The central technical result is that **Hermes Self-Evolution and the Civilization's Capability Evolution chain describe the same loop** — but Hermes terminates at "open a PR," with its safety leash living in a human reviewer, whereas the Civilization made the leash a fail-closed property of the substrate (e.g. *"optimizer actor cannot promote its own output,"* enforced in `eventgraph/.../capability_evolution.go`). Six concrete, implementable learning opportunities are identified, all consistent with the existing deferral: they describe how to *use Hermes as the reference optimizer inside a chain the Civilization already owns*, not how to import it as authority.

## 1. Introduction and Scope

This evaluation was commissioned to determine whether the Hermes Agent family represents a comparable capability to the Transpara-AI Civilization, whether the Civilization can derive actionable learnings from Hermes's published implementation, and whether the current released baseline changes any conclusion reached during the 2026-05-13 landscape investigation.

The evaluation covers:

- **`hermes-agent`** — the self-improving agent runtime ("the agent that grows with you"), forked from `NousResearch/hermes-agent` on 2026-05-11.
- **`hermes-agent-self-evolution`** — the DSPy+GEPA evolutionary optimizer that operates *on* `hermes-agent`, forked 2026-05-11.
- **`hermes-example-plugins`** — reference plugins demonstrating the plugin boundary, forked 2026-05-13.
- The prior canonical adjudication: the v3.9.1 External Technology Decision Crosswalk, Decision 15, ADR-0010, and update items U7/U8.
- The native equivalent that was built as a result: the EventGraph Capability Evolution chain (`transpara-ai/eventgraph`, `darkfactory/v39` package).

**Evidence base and source discipline.** Per the standing rule that *code and canonical repo contents are ground truth, while marketing pages may be aspirational*, this evaluation synthesizes from (a) the compiled canonical wiki articles on Hermes Agent, Hermes Self-Evolution, and Capability Evolution; (b) the investigation's Phase-4 Batch C/D checkpoints (README/PLAN-grounded); (c) ADR-0010; and (d) direct reading of the `transpara-ai/eventgraph` Go implementation of the Capability Evolution chain. The upstream `NousResearch` repositories are cited only as context for *why the org forked them to read them* — they are not re-published. Claims about Hermes internals are README/PLAN-grounded and are flagged where confidence is thin; claims about the native chain are code-anchored.

The Transpara-AI Civilization reference baseline is the same one used in TAI-RES-2026-001, taken at HEAD as of 2026-06-24, with one addition relevant to this subject: the v3.9 Capability Evolution chain implemented in the `eventgraph` repo.

## 2. Hermes Agent & Hermes Self-Evolution: Capability Overview

Hermes is an open-source project family from **Nous Research**, pulled into the `transpara-ai` org as three forks so the code could be inspected as source. "Hermes" in this document names that cluster — not a single product. The investigation deliberately worked at the resolution of three repos with three distinct roles; the decision crosswalk later collapsed them into one "Hermes" row. Both layers are kept visible below.

### 2.1 The Three Forks

| Repo | Role | Investigation verdict | License |
|---|---|---|---|
| `hermes-agent` | Self-improving agent runtime — terminal, messaging, learning loop, skills, memory, cron, subagents, multi-backend execution | Optional future **runtime adapter** (RuntimeBroker candidate); not Base Slice 0 | MIT |
| `hermes-agent-self-evolution` | Offline DSPy+GEPA optimizer that evolves Hermes's skills/prompts/tools/code and opens PRs | Optional **capability-evolution** pattern; "high-priority governed optimizer pattern; no activation authority" | MIT (with an AGPL-v3 code-evolution path, see §2.4) |
| `hermes-example-plugins` | Reference plugins demonstrating the plugin surface | **Pattern / reference only** for plugin/capability policy | MIT |

### 2.2 Hermes Agent — the Self-Improving Runtime

Described by the upstream as *"a self-improving AI agent from Nous Research"* and tagged *"the agent that grows with you,"* `hermes-agent` bundles a broad runtime surface: a terminal interface, a messaging gateway, a built-in learning loop, a skills system, persistent memory and user modeling, a cron scheduler, subagents, RPC/scripts, model-provider flexibility, batch trajectory generation, and RL/training integrations.

Its distinguishing strength is **execution-backend breadth.** It can run against local, Docker, SSH, Singularity, Modal, Daytona, and Vercel Sandbox backends, and supports migration from OpenClaw. The investigation judged it the *strongest agent-execution candidate for a future runtime adapter* precisely because it already addresses skills, memory, terminal backends, cron, subagents, migration, and trajectory generation in one package.

What it explicitly **is not** (per the Batch C analysis): it is not an EventGraph truth source, a Work DAG, an authority model, a release-certification authority, or a capability-governance system. Its command-approval, DM-pairing, and container-isolation surfaces are *not* the Civilization's AuthorityRequest/AuthorityDecision/ExecutionReceipt primitives, and its built-in learning loop is *not* sufficient for governed capability promotion.

> ⚠ **Evidence note.** All `hermes-agent` internals above are **README-grounded only** — the landscape investigation inspected the README, not the code. Treat the feature list as the upstream's self-description, corroborated across the org's notes but not code-verified.

### 2.3 Hermes Self-Evolution — the DSPy/GEPA Optimizer

This is the repo the evaluation's theme foregrounds. `hermes-agent-self-evolution` is a **standalone optimization pipeline that operates *on* `hermes-agent`, not inside it.** It reads execution traces, proposes targeted textual mutations to skills/prompts/tools, evaluates candidate variants against test tasks, applies constraint gates, and opens a PR for human review. It runs **entirely via API calls — no GPU training, no weight updates.** All engines evolve and select *text* (skill files, tool descriptions, prompt sections, code files), never model weights; DSPy's weight-training component (`BootstrapFinetune`) is explicitly excluded.

**Three engines, one workflow:**

| Engine | Targets | License | Status in plan |
|---|---|---|---|
| **DSPy + GEPA** (Genetic-Pareto Prompt Evolution) | Skill files, tool descriptions, prompt sections | MIT | Primary — fully integrated |
| **DSPy MIPROv2** | Few-shot examples, instruction text | MIT | Fallback optimizer |
| **Darwinian Evolver** (imbue-ai) | Code files, algorithms, tool implementations | **AGPL v3** | External CLI only — Phase 4 |

GEPA (reported as an ICLR 2026 Oral) is the lead engine: it integrates into DSPy, reads execution traces to understand *why* a candidate fails rather than merely that it failed, and is claimed to converge with as few as three examples and to outperform both RL and prior DSPy optimizers on prompt-optimization benchmarks. *(These performance claims trace to the upstream README and are carried as medium confidence, not independently verified.)*

**The optimization loop**, quoted from the README:

```
Read current skill/prompt/tool
Generate eval dataset
Run GEPA optimizer using execution traces
Produce candidate variants
Evaluate
Apply constraint gates
Open PR against hermes-agent
```

**The five-tier staging plan** (only Tier 1 implemented as of the last commit, 2026-03-29):

| Tier | Target | Engine | Status |
|---|---|---|---|
| 1 | Skill files (`SKILL.md`) | DSPy + GEPA | Implemented |
| 2 | Tool descriptions | DSPy + GEPA | Planned |
| 3 | System-prompt sections | DSPy + GEPA | Planned |
| 4 | Tool implementation code | Darwinian Evolver | Planned |
| 5 | **Continuous improvement loop** | Automated pipeline | Planned |

**Five guardrails** — every evolved variant must pass all five before a PR is opened: (1) the full test suite (`pytest tests/ -q`) at 100%; (2) size limits (skills ≤ 15 KB, tool descriptions ≤ 500 chars); (3) caching compatibility (no mid-conversation changes); (4) semantic preservation (no drift from original purpose); (5) PR review (human review; no direct commits). The pipeline opens a PR against `hermes-agent`; it never writes directly to the branch.

**Evaluation data** comes in two modes: `--eval-source synthetic` (dataset generated from the artifact under test) or `--eval-source sessiondb` (real session history imported from Claude Code, Copilot, or Hermes — higher signal). A session importer was added in the final commit, confirming the `sessiondb` path is implemented for Tier 1.

**Estimated cost:** ~$2–10 per optimization run at Tier-1 scope (skill evolution, 10 iterations); costs scale with tier depth and iteration count.

### 2.4 Upstream Identity and Licensing

The two named repos were forked from `NousResearch/hermes-agent` and `…/hermes-agent-self-evolution` on 2026-05-11; `hermes-example-plugins` on 2026-05-13 (confirmed via live GitHub fork metadata, 2026-06-13). `hermes-agent` and `hermes-example-plugins` are MIT. The self-evolution repo's README declares MIT, **but its PLAN flags a Darwinian Evolver (imbue-ai) code-evolution path under AGPL v3 that must remain "external CLI only."** If Tier 4 (code evolution) is ever activated, that AGPL boundary is a supply-chain gate, not a footnote.

## 3. Transpara-AI Civilization: Baseline Architecture

The Transpara-AI Civilization is a self-organizing AI agent civilisation built on the EventGraph substrate. It is not a product — it is a platform for collective existence whose goal is to build products that fund the agents that build more products, on a transparent, auditable causal chain. (This section reuses the TAI-RES-2026-001 baseline so the two evaluations stay parallel, and adds the Capability Evolution chain, which is the native machinery most relevant to a Hermes comparison.)

### 3.1 Architecture Layers

| Layer | Description |
|---|---|
| EventGraph Substrate | 201 primitives. Causal chain where every action, resource, and outcome is linked. Trust model, authority levels (Required → Recommended → Notification), cryptographic consent. All on Postgres on-prem. |
| Hive Runtime | 4 bootstrap agents: CTO, Guardian, Researcher, Planner/Implementer. FSM-based agentic loop (observe → reason/operate → check stopping → repeat). Go binary. Deterministic agent identity. |
| Social Grammar | Constitutional law for agents. 10 invariants. 8 formal agent rights (existence, memory, identity, communication, purpose, dignity, transparency, boundaries). Dual human-agent consent required to amend. |
| Decision Tree Evolution | Agents learn within their lifetime: LLM-requiring decisions track response history. After ≥50 hits at ≥95% accuracy, deterministic branches replace LLM calls. The mechanical-to-intelligent continuum. |
| **Capability Evolution Chain** | Governed self-improvement of *shared* capabilities (skills, plugins, prompt sections, tool descriptions, runtime adapters, policy bundles). A fixed sequence of typed records implemented in `transpara-ai/eventgraph`. Detailed in §3.3. |
| 13 Product Layers | Work, Market, Social, Justice, Build, Knowledge, Alignment, Identity, Bond, Belonging, Meaning, Evolution, Being. Each runs on the same graph; each makes the graph more valuable. |
| Self-Sustaining Economy | Revenue funds agents. Agents build products. Products generate revenue. Every resource in and out on the causal chain, publicly auditable. Corporations pay; individuals free. |

### 3.2 Role Architecture (abridged)

The Civilization grows its own workforce through the Spawner role. Roles derive from four dimensions: domain, response type, time horizon, and model intelligence tier (Opus/Sonnet/Haiku). Bootstrap roles (Tier A) exist from day one; Tier B–D roles emerge as the system proves it needs them. Relevant to this evaluation: a `CapabilityReviewer`/`CapabilityReleaseRole` actor is the *only* role permitted to promote a capability version, and the RoleArchitect (Tier D) evolves role specs over time.

### 3.3 The Capability Evolution Chain (the native equivalent of Hermes Self-Evolution)

The Civilization treats every change to a shared capability as **governed production work flowing through a chain of evidence**, not as an autonomous optimization loop. The doctrine's one-line intent: *"Capability evolution is production work, not an uncontrolled self-improvement loop."* The chain is a fixed sequence of ten typed records, all implemented as first-class types in `eventgraph/go/pkg/darkfactory/v39/schema.go`:

```
EvolutionOrder → EvalDataset → OptimizationRun → CandidateVariant →
BenchmarkResult → HumanReview → CapabilityVersion → ActivationPolicy →
FactoryRuntimeVersion → RollbackRecord (when needed)
```

The load-bearing property is **what the optimizer is forbidden to do**, and in this implementation those prohibitions are enforced at the store boundary, not in helpers (code references from `capability_evolution.go`):

- **An optimizer cannot promote its own output.** `if run.CreatedBy == version.PromoterActorID || candidate.CreatedBy == version.PromoterActorID { … "optimizer actor cannot promote its own output" }` (≈L289–290).
- **Only the release role may promote.** `if version.PromoterRole != CapabilityReleaseRole { … }` (≈L238).
- **Global activation is disabled.** `if policy.Scope == "global" { … "global activation disabled for MVP" }` (≈L143–144).
- **Human review cannot be forged loosely.** The promoting `HumanReview` *"must include promoted CandidateVariant and BenchmarkResult"* (≈L341); an approved candidate (≈L275), an accepted evolution order (≈L302), and an active promoter identity (≈L352) are all required.
- **Activation is a property of an edge, not a writable field.** `"active"` is not a persistable status; activation is the existence of an `ACTIVATED_BY` edge written through a trusted-only path. A follow-up bound promotion to the authority chain (`AuthorityRequest → AuthorityDecision → ExecutionReceipt`).

The MVP is deliberately **text-only** (skills, tool descriptions, prompt sections) and excludes runtime-code mutation, tool-implementation mutation, policy changes, global activation, auto-merge, automatic activation, and continuous loops. Capability evolution is also forbidden in Base Slice 0 — it is a capability the factory *earns*, not a starting condition.

> ⚠ **Fail-legible note.** The "optimizer cannot weaken gates / self-promote" property is *earned, not free*. The first implementation placed the gates in helpers and let direct appends bypass them; the property became true only after two-plus adversarial-review rounds moved the checks to the store boundary and a third round still found authority bypasses. The model is sound; making it fail-closed in code took iteration. This is exactly the asymmetry that makes the chain a deeper artifact than an optimizer that simply "opens a PR."

## 4. Comparative Analysis

The central finding is that Hermes and the Transpara-AI Civilization are **complementary systems operating at adjacent layers** — but with a far closer technical correspondence than Sakana had. Hermes Self-Evolution is, almost line for line, a *reference implementation of the loop the Civilization already governs in code.*

### 4.1 Capability Comparison Matrix

| Dimension | Hermes (Agent + Self-Evolution) | Transpara-AI Civilization | Verdict |
|---|---|---|---|
| Primary output | A self-improving agent + an optimizer that evolves it | Self-sustaining AI-built products | Different goals |
| Self-improvement mechanism | DSPy+GEPA evolves skills/prompts/tools/code; opens PR | Capability Evolution chain: 10 typed records, code-enforced leash | **Same loop; Civilization governs it** |
| Optimizer authority boundary | Convention: pipeline "opens a PR," never commits | Code: optimizer cannot promote own output; only release role promotes | **Civilization enforces in substrate** |
| Eval data | Synthetic or SessionDB trace mining | EvalDataset fixed *before* optimization; governed as evidence | Similar intent, Civilization governs provenance |
| Agent runtime | Multi-backend (Docker/SSH/Modal/Daytona/Vercel), cron, subagents | Hive runtime + RuntimeBroker adapter boundary (RuntimeEnvelope) | **Hermes leads on backend breadth** |
| Plugin/skill model | `hermes-example-plugins` reference surfaces | CapabilityArtifact governance (provenance, eval, rollback) | Similar surface; Civilization governs lifecycle |
| Activation scope control | None in pipeline (PR merge is the act) | ActivationPolicy: order/project/canary; "global" rejected in code | **Civilization unique** |
| Rollback | Git revert (external) | RollbackRecord as a first-class chain stage; autonomy gated on rollback | **Civilization deeper** |
| Governance | Human PR review only | Decision 15, authority chain, dual-consent constitution | **Civilization unique** |
| Causal accountability | Trace logs + PR diffs | EventGraph causal chain: every resource auditable | **Civilization deeper** |
| Agent identity / rights | Agents/skills are artifacts to optimize | 8 formal rights, persistent identity, memory | **Civilization unique** |
| Continuous self-improvement loop | Tier 5 (planned) | **Forbidden** as a production loop (Decision 15 / MVP exclusion) | Deliberate divergence |
| Maturity of the optimizer engine | GEPA: trace-reflective, ~3-example convergence | Native chain has the *governance*, not yet a best-in-class engine | **Hermes leads on engine** |

### 4.2 Shared Territory

Hermes and the Civilization converge on the *same* problem more tightly than Sakana did:

- **Governed self-improvement.** Hermes's loop (read → eval dataset → optimize → candidate → evaluate → gate → PR) and the native chain (EvolutionOrder → EvalDataset → OptimizationRun → CandidateVariant → BenchmarkResult → HumanReview → …) are the same pipeline. The investigation noted this explicitly: Hermes's loop *"already maps onto"* the native EvolutionOrder/CandidateVariant/BenchmarkResult/HumanReview/ActivationPolicy/RollbackRecord chain.
- **Trace-driven evaluation.** Hermes mines execution traces (SessionDB) for eval data; the Civilization fixes an EvalDataset from held evidence before optimization. Same insight: measure candidates against real behavior, not vibes.
- **Text-first, low-blast-radius scope.** Hermes Tier 1 is skill files; the native MVP is text-only (skills, tool descriptions, prompt sections). Both chose the same safe entry point independently.
- **Capabilities as governed artifacts.** `hermes-example-plugins` treats plugins as first-class, copyable surfaces; the Civilization treats skills/plugins/prompt sections as CapabilityArtifacts deserving provenance and rollback.

### 4.3 The Civilization's Structural Advantages

These have no Hermes equivalent and represent the deepest difference:

- **The leash is in the substrate, not the reviewer.** Hermes's safety rests on the convention that the pipeline "opens a PR" and a human reviews it. The Civilization makes "an optimizer cannot promote its own output" a compile-checked predicate at the store boundary, with activation scope, promoter role, and authority binding all enforced in code.
- **Activation and rollback are first-class.** Hermes ends at the PR. The Civilization continues: ActivationPolicy (canary/project scope, never global), FactoryRuntimeVersion (a bill of materials), and a RollbackRecord stage — with the platform invariant that *autonomy may not expand unless rollback expands with it.*
- **Causal accountability.** Every EvolutionOrder, candidate, benchmark, and activation is a linked, auditable EventGraph record. Hermes produces trace logs and diffs, not a causal chain.
- **The "no continuous loop" rule is principled, not incidental.** Hermes's Tier 5 (continuous improvement loop) is precisely what the Civilization forbids as a production loop. The Civilization didn't fail to build it — it decided, by doctrine, that it must not run before the governing controls exist.
- Plus the Civilization-wide advantages shared with the Sakana analysis: constitutional agent rights, the self-sustaining economy, and 13-layer civilisational scope.

### 4.4 Reconciliation With Prior Canonical Decisions

Unlike Sakana, Hermes already has a paper trail this evaluation must stay consistent with — or explicitly update. It does not update it; it confirms it against the current released baseline.

| Prior decision | Source | This evaluation's finding |

| "Governed optimizer pattern only / deferred; must not become a continuous production loop in v3.9; Epic 8+" | v3.9.1 Decision Crosswalk (Hermes row) | **Confirmed.** Nothing in the current baseline changes it; Tier 5 remains forbidden as a production loop. |
| Three forks, three decisions (runtime adapter / plugin-reference / capability-optimizer) | Phase 5 inclusion matrix | **Confirmed and preserved** (§2.1, §5). The single crosswalk row remains a coarse collapse. |

## 5. Strategic Learning Opportunities

Six concrete, implementable opportunities. Crucially, none requires new infrastructure or a reversal of the deferral: each describes how to *use Hermes as the reference optimizer/adapter inside the chain the Civilization already owns.*

### 5.1 GEPA → the Engine for the `OptimizationRun → CandidateVariant` Stage

**Current state:** The native Capability Evolution chain has the *governance* (ten typed records, code-enforced leash) but no specified best-in-class optimizer for the one stage an optimizer is allowed to own: `OptimizationRun → CandidateVariant`.

**Hermes pattern:** GEPA reads execution traces to understand *why* a candidate fails and converges with as few as three examples — a reflective, sample-efficient search rather than blind mutation.

**Learning:** Adopt GEPA (or DSPy+GEPA) as the engine that *produces CandidateVariants* inside an EvolutionOrder. This is the safest possible adoption: the optimizer's authority already ends at CandidateVariant by code (`capability_evolution.go` rejects self-promotion at ≈L289–290), so plugging in a stronger engine cannot widen its blast radius. The leash is pre-installed; this just upgrades the thing on the leash. Scope it to text-only Tier-1 targets (skills, tool descriptions) to match the MVP.

### 5.2 SessionDB Trace Mining → Governed `EvalDataset` Generation

**Current state:** The chain requires an EvalDataset fixed *before* optimization, but the source of that dataset is unspecified.

**Hermes pattern:** `--eval-source sessiondb` imports real session history (Claude Code, Copilot, Hermes) as high-signal evaluation data; a session importer is already implemented.

**Learning:** Use real-trace mining to populate EvalDataset — but route it through governed memory/evidence, not raw private context. Each imported trace becomes a MemoryReference/KnowledgeReference with influence-recording, redaction state, and risk scope, exactly as the Civilization's memory-knowledge model requires. This captures Hermes's biggest quality lever (real traces beat synthetic data) while closing its biggest risk (SessionDB mining can leak or overfit private data). EvalDataset stays fixed before optimization so the target cannot move to fit the result.

### 5.3 Hermes's Five Guardrails → Confirm/Complete the Native Gates

**Current state:** The native gates enforce evidence integrity, promoter role, authority binding, and non-global activation. Two of Hermes's guardrails have no explicit native predicate yet.

**Hermes pattern:** (1) full test suite at 100%, (2) size limits, (3) caching compatibility (no mid-conversation changes), (4) semantic preservation, (5) PR review.

**Learning:** Guardrails 1, 4, and 5 already correspond to BenchmarkResult, the eval gate, and HumanReview. The two cheap additions are **size limits** and **caching compatibility** — both are pure validation predicates that belong in `validate.go` alongside the existing status allowlists, costing a few lines each. Adopting them makes the native chain a strict superset of Hermes's guardrail set.

### 5.4 Hermes Agent Backends → the RuntimeBroker Adapter Reference (deferred, Epic 6+)

**Current state:** The Civilization specifies a RuntimeBroker adapter boundary (RuntimeEnvelope, protected-action policy, a 10-point conformance gate) but has no reference adapter selected.

**Hermes pattern:** `hermes-agent` runs against local, Docker, SSH, Singularity, Modal, Daytona, and Vercel Sandbox backends, with cron and subagents — the broadest backend surface in the surveyed landscape.

**Learning:** Keep `hermes-agent` as the **reference design for the RuntimeBroker adapter**, behind the conformance gate (file/command/network/secret boundaries, timeout, artifact capture, nonzero-exit behavior, EventGraph receipt emission, post-run validation, replayability). This stays deferred to Epic 6+ per the existing decision; the learning is to mine its backend abstraction, not to integrate it.

### 5.5 hermes-example-plugins → CapabilityArtifact Governance (U7)

**Current state:** U7 established that skills/plugins/workflow packs are CapabilityArtifacts; the lifecycle governance is still maturing.

**Hermes pattern:** The example plugins demonstrate host-owned structured LLM calls, async LLM access, and dashboard tabs/routes — and are explicitly *not bundled*, so authors copy what they want. The named risk is "ungoverned plugins."

**Learning:** Use the plugin surface as the concrete checklist for what CapabilityArtifact governance must cover (provenance, eval, activation scope, rollback) so that a copied plugin cannot enter the factory without becoming a governed artifact. This is reference-only adoption — exactly its Phase-5 classification.

### 5.6 ADR-0010 → Execute the First Governed EvolutionOrder

**Current state:** ADR-0010 ("governed capability optimizer") is *Proposed*; its implementation note says to "target a single low-to-medium-risk skill or tool description" first, with code evolution deferred until full test and benchmark gates exist.

**Hermes pattern:** Tier 1 (skill evolution) is the implemented, lowest-risk, directly measurable entry point, at ~$2–10 per run.

**Learning:** Stand up **one** concrete Epic-8 EvolutionOrder end-to-end: pick a single low-risk skill, generate an EvalDataset from governed traces (§5.2), produce CandidateVariants with GEPA (§5.1), score a BenchmarkResult, route a HumanReview by the CapabilityReviewer, promote a CapabilityVersion, activate at canary scope, and keep a RollbackRecord. This is the minimal "earn it" step that turns the built-but-unused chain into a live, governed loop — and it is the natural moment to move ADR-0010 from *Proposed* to *Accepted*. Tier 4 (code evolution) stays out until the AGPL Darwinian Evolver boundary (§2.4) is cleared.

## 6. The Inverse: What Hermes Cannot Learn From the Civilization

| Civilization Capability | Why Hermes Has No Equivalent |
|---|---|
| The governed Capability Evolution chain | Hermes's optimizer ends at "open a PR." It has no EvolutionOrder, ActivationPolicy, FactoryRuntimeVersion, or RollbackRecord, and no authority binding. Its leash lives in a human reviewer; the Civilization's lives in the substrate as code-enforced predicates. Hermes cannot replicate this without building the chain. |
| EventGraph causal chain | Hermes produces trace logs and PR diffs. There is no substrate linking optimizer runs to resource consumption to activation to rollback to public auditability. |
| Code-enforced "optimizer cannot self-promote" | In Hermes this is a convention (the pipeline doesn't merge). In the Civilization it is a predicate at the store boundary (`capability_evolution.go` ≈L289–290), proven over the input domain after adversarial review. |
| Activation-scope governance | Hermes has no notion of canary/project/global activation; a merged PR is simply live. The Civilization rejects "global" scope in code and treats activation as a governed edge. |
| Constitutional agent rights | Hermes treats agents and skills as artifacts to optimize. There is no agent identity persistence, memory rights, or dignity. |
| Self-sustaining economy | Hermes is an OSS pipeline with a per-run API cost. The Civilization's design funds agents from the products agents build. |
| Civilisational scope (13 layers) | Hermes is an agent + optimizer. The Civilization spans Work, Justice, Identity, Bond, Belonging, Meaning, Evolution, and Being. |
| "No continuous production loop" as doctrine | Hermes's roadmap *aspires* to a continuous Tier-5 loop. The Civilization forbids it by doctrine until audit and rollback can keep pace — autonomy gated on reversibility. |

## 7. Conclusions

Hermes is the most *technically convergent* external system the Civilization has evaluated: its self-evolution pipeline is, almost stage for stage, a reference implementation of the loop the Civilization already governs. That convergence is why the earlier landscape investigation called it "high-priority" while still deferring it — and why this evaluation, against the current released baseline, **confirms every prior decision** rather than reopening any. The difference between the two systems is not the loop; it is *where the leash lives.* Hermes places it in a human reviewer at PR time; the Civilization placed it in the substrate, as fail-closed predicates that took multiple adversarial rounds to make true in code. That is the Civilization's structural advantage, and it is exactly the advantage Hermes cannot adopt without rebuilding.

**Priority recommendations:**

1. **Highest priority:** Adopt GEPA as the engine for the `OptimizationRun → CandidateVariant` stage (§5.1). The optimizer's authority is already bounded in code, so a stronger engine adds quality with no added blast radius. Text-only scope.
2. **High priority:** Execute one end-to-end governed EvolutionOrder on a single low-risk skill (§5.6), and move ADR-0010 from *Proposed* to *Accepted*. This activates the built-but-idle chain under full governance.
3. **Medium priority:** Govern SessionDB-style trace mining as EvalDataset/MemoryReference with influence-recording and redaction (§5.2); add the two missing guardrails (size limits, caching compatibility) as native validation predicates (§5.3).
4. **Medium priority (deferred, Epic 6+):** Keep `hermes-agent` as the reference RuntimeBroker adapter design behind the conformance gate (§5.4).
5. **Lower priority:** Use `hermes-example-plugins` as the checklist for CapabilityArtifact/plugin governance under U7 (§5.5).

**Standing constraints reaffirmed:** Tier 5 (continuous production loop) remains forbidden as a production loop under Decision 15. Tier 4 (code evolution via the AGPL-v3 Darwinian Evolver) must clear the supply-chain/license boundary before any adoption. All learnings keep Hermes as engine, adapter, or pattern — never as kernel, promoter, release authority, or factory controller.
